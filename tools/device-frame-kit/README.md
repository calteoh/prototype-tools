# Device Frame Kit

Show your mobile-web prototype inside a realistic iPhone frame — on any desktop browser.

Built for design reviews: present from your laptop, screen-share on calls, or send a link so people can experience a mobile-first design without picking up their phone. On an **actual phone, none of this appears** — the prototype renders normally, full-screen.

What you get around your prototype:

- **A phone frame** (iPhone 14/15 size: 390 × 844) centered on the page
- **A simulated iOS status bar** — 9:41, Dynamic Island, signal/wifi/battery
- **A simulated iOS 26 "Liquid Glass" Safari bar** — floating glass pills (back · search · more) over a contrast fade that the page scrolls *underneath*; they collapse to a single compact domain pill when you scroll down and expand when you scroll up, reach the top, or tap it
- **A "Pages" menu** (hamburger button) to jump straight to any page or state of your prototype
- **A QR code button** so reviewers can open the prototype on their own phone

Everything is switched on/off in **one config file**. There is no settings UI on screen — what reviewers see is just your prototype in a phone.

## What you need

> **Prerequisites — bring these:**
> 1. **Node.js 20** (or newer) installed — [nodejs.org](https://nodejs.org)
> 2. **pnpm or npm** (pnpm recommended; npm works the same)
> 3. **A Next.js project using the App Router** (it has an `app/` folder with a `layout.tsx`). If you're starting fresh, this kit folder *is* a working Next.js project — copy the whole thing and build inside it.
>
> No other libraries needed. The kit has **zero dependencies** beyond React/Next themselves — even the QR code is generated by code inside the folder.

## Add it to your project — 3 steps

**1. Copy the `device-frame/` folder** into the root of your Next.js project (next to your `app/` folder). The folder is self-contained — components, styles, config, everything.

**2. Wrap your layout.** In `app/layout.tsx`, import the component and wrap your page content:

```tsx
import { DeviceFrame } from "../device-frame/DeviceFrame";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <DeviceFrame>{children}</DeviceFrame>
      </body>
    </html>
  );
}
```

**3. Edit `device-frame/device-frame.config.ts`.** That's the whole control panel:

```ts
export const config: DeviceFrameConfig = {
  ...defaultConfig,

  device: "standard",          // "standard" (390×844) | "large" (430×932) | { width, height }
  frame: true,                 // false = raw viewport, no shell/corners/shadow
  backdrop: "#f5f5f4",         // canvas color behind the phone

  statusBar: true,             // simulated iOS status bar
  dynamicIsland: true,         // false = hide notch, keep status-bar indicators
  browserChrome: true,         // simulated Safari bottom bar
  urlBarDomain: "yourproject.com", // the domain shown in the URL pill

  qrCode: true,                // QR button (top right)
  pageNav: true,               // Pages drawer (top left)
  pages: [
    { label: "Home", path: "/" },
    { label: "Product page", path: "/product" },
  ],

  simulateTouch: false,        // mouse acts as a finger (see below)
};
```

Run your project, open it on a desktop browser — your prototype is in the phone.

## Registering pages and states

The Pages drawer exists so reviewers can jump straight to a screen **without clicking through the flow**. Each entry is just a URL — every click does a fresh page load inside the phone, so the drawer never needs to know anything about your routing.

```ts
pages: [
  { label: "Home", path: "/" },
  { label: "Checkout — empty", path: "/checkout?state=empty", section: "Checkout" },
  { label: "Checkout — error", path: "/checkout?state=error", section: "Checkout" },
],
```

- `section` groups entries under a small header (optional).
- **States via query params:** if a screen has multiple states, let the page read a query param (e.g. `useSearchParams().get("state")`) and register each state as its own entry. Any URL your prototype can boot from is a valid jump target — query params, hash links, anything.
- The demo's `app/detail/page.tsx` shows this pattern working.

## The simulated Safari bar — what your design should know

The bar is modeled on **iOS 26's "Liquid Glass" Safari**: floating glass pills (a back button, a center search bar, and a more button) that hover above your page, with a soft contrast fade behind them down to the bottom edge. Your content scrolls **underneath** them (the glass blurs whatever is behind it), and they collapse to a single compact domain pill after ~30px of downward scrolling — any scrollable area counts, not just the page — then expand when you scroll up, hit the top, or tap the pill. The contrast fade is only present in the expanded state.

Because the bar floats *over* the content (rather than the viewport ending above it), your design gets the full screen height; the bar overlays the bottom, exactly like real iOS 26.

Two hooks your CSS can use:

**1. Safe areas.** The status bar overlays the top 59px and the floating bar occupies the bottom. Pad your fixed chrome with the kit's variables:

```css
padding-top: var(--df-safe-area-top, env(safe-area-inset-top));
padding-bottom: var(--df-safe-area-bottom, env(safe-area-inset-bottom));
```

`--df-safe-area-top` is `59px` (status bar). `--df-safe-area-bottom` **tracks the URL bar's height** — `82px` when expanded, `46px` when scrolled — so a fixed bottom element (a CTA, a tab bar) padded by it rides up and down *with* the bar as it collapses. The value animates smoothly on its own (registered via `@property`), so you don't need to add a transition. Both are `0px` when you switch the chrome off, and on a real phone the variables don't exist so the `env()` fallbacks take over — the same CSS works everywhere.

**2. React to the collapse.** When the Safari bar minimizes, the kit sets `data-df-chrome="collapsed"` on your prototype's `<html>`:

```css
[data-df-chrome="collapsed"] .my-bottom-bar { /* tweak styles when the bar is small */ }
```

Treat the expanded state as your default — the attribute only appears after the first scroll.

## Touch simulation (`simulateTouch`)

Off by default. Turn it on if your prototype has touch-only interactions (drag carousels, swipe gestures): the mouse then behaves like a finger inside the phone — it fires real `touchstart` / `touchmove` / `touchend` events, suppresses the phantom click after a drag, reports the device as a touchscreen (`pointer: coarse`), and replaces the cursor with a round fingertip that reads well on screen shares.

Leave it off if your prototype changes behavior based on pointer type — with it on, your code will believe it's running on a touchscreen.

## Sharing and deploying

- **QR code:** the button in the top-right encodes the current page's plain URL (without the internal `embed` parameter). Deploy anywhere (Netlify, Vercel), open the prototype, click the QR — reviewers scan it and get the frameless mobile experience on their phone.
- **Deploying under a sub-path** (e.g. `yoursite.netlify.app/my-prototype/`): build with the base path set —

  ```bash
  NEXT_PUBLIC_BASE_PATH=/my-prototype pnpm build
  ```

  The kit's navigation and QR codes handle the sub-path automatically. **But note:** Next.js does *not* rewrite plain image/video `src` strings. If your prototype references files in `public/` (like `<img src="/photos/bag.jpg">`), prefix them with a tiny helper or they will 404 on the deployed build:

  ```ts
  // lib/asset.ts
  const base = process.env.NEXT_PUBLIC_BASE_PATH ?? "";
  export function asset(path: string): string {
    if (!path || !path.startsWith("/")) return path;
    return `${base}${path}`;
  }
  ```

  Then use `<img src={asset("/photos/bag.jpg")} />`.

## Troubleshooting

| Symptom | Cause / fix |
| --- | --- |
| **Images work locally but 404 when deployed** | You're deployed under a sub-path and raw `src` strings aren't prefixed. Use the `asset()` helper above. |
| **`?embed=1` appears in the URL** | That's the kit's internal marker for "this copy runs inside the phone iframe." Harmless. Don't link to it directly; share the plain URL. |
| **The frame shows up on my phone/tablet** | The desktop detection saw a mouse-like pointer (some tablets with trackpad keyboards do this). Real phones always pass through. Scan the QR code instead of typing the URL — same result. |
| **My drag/swipe interaction doesn't work in the frame** | Enable `simulateTouch: true` in the config. |
| **Touch simulation breaks my pointer-type logic** | Expected — `simulateTouch` makes the prototype believe it's a touchscreen. Turn it off, or gate your logic with the kit's `useEmbedded()` hook. |
| **Two scrollbars / desktop scrollbars inside the phone** | The kit hides them automatically; if you still see one, check for `overflow` set on a parent of `<html>`-level content. |
| **A fixed bottom element is hidden behind the floating bar** | Expected at rest — the iOS 26 bar overlays bottom content. Pad it with `var(--df-safe-area-bottom, env(safe-area-inset-bottom))` to clear the home indicator, and add extra margin if it must stay clear of the resting bar. |
| **The floating bar looks opaque / doesn't blur the page** | The glass uses `backdrop-filter`. It works in current Chrome/Safari; if you've disabled hardware acceleration or use an old browser, the blur may not render — present in an up-to-date Chromium browser. |

## What's in the box

```
device-frame/
├── device-frame.config.ts   ← the only file you edit
├── DeviceFrame.tsx          ← the only component you import
├── useEmbedded.ts           ← optional hook: "am I inside the frame?"
├── components/              ← status bar, Safari bar, drawer, QR (internal)
├── hooks/                   ← frame detection, Safari state machine (internal)
├── lib/                     ← iframe bridge, touch synthesis, QR encoder (internal)
└── styles/                  ← scoped CSS (no Tailwind required)
```

Currently iOS/Safari only. The config's `device` field accepts custom dimensions (`{ width, height }`) for other screen sizes; other OS chromes (Android/Chrome) may come later.
